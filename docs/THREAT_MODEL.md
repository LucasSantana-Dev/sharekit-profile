# Threat Model — sharekit-profile

> Committed security artifact. Covers the primary attack surfaces of the operator harness: credential exposure, shell injection, SSRF, path traversal, MCP scope creep, AI attribution leakage, and PR automation bypass.

---

## 1. Credential Exposure (API Keys, Tokens)

**Threat:** Secrets (API keys, OAuth tokens, SSH keys) leak into git history, logs, or agent context.

**Attack Vector:**
- Agent reads `.env`, `~/.ssh/*`, or `~/.aws/credentials` and includes content in output
- Hook scripts log environment variables containing secrets
- RAG index ingests files containing hardcoded credentials
- Session transcripts saved to `~/.claude/transcripts/` contain tokens from tool output

**Mitigation:**
- `PreToolUse` hook blocks reads to `~/.ssh`, `~/.aws`, `/etc` protected paths
- `.gitignore` excludes `.env`, `*.pem`, `credentials.json`, and similar patterns
- RAG indexer excludes secret-pattern files (`.env`, `secrets/`, `credentials/`)
- `mcp-policy.json` sets `defaultDeny: true` — MCP servers must be explicitly approved
- Dangerous pattern regex in `mcp-policy.json` catches `curl` with inline tokens

**Severity:** Critical

---

## 2. Shell Injection via Hooks

**Threat:** Malicious input in hook scripts leads to arbitrary command execution.

**Attack Vector:**
- Hook reads untrusted input (file content, user prompt) and interpolates it into a shell command without quoting
- `settings.json` hook definitions reference scripts in writable locations that an attacker could replace
- RAG recall output injected into shell context via `UserPromptSubmit` hook

**Mitigation:**
- All hook scripts use `"$variable"` quoting (never unquoted interpolation)
- Hooks live in `~/.claude/hooks/` (user-owned, not world-writable)
- Hook timeout enforced at 3-5s — limits blast radius of runaway scripts
- `PreToolUse` dangerous pattern filter blocks `curl|bash`, `sudo rm`, and similar before they execute
- `mcp-policy.json` sets `allowShell: false` by default

**Severity:** High

---

## 3. SSRF via Web Tools

**Threat:** Agent uses `webfetch` or MCP `fetch` server to access internal network resources (metadata endpoints, localhost services, internal APIs).

**Attack Vector:**
- Prompt injection in a webpage causes the agent to fetch `http://169.254.169.254/` (cloud metadata)
- Agent follows a redirect from an external URL to an internal service
- MCP `fetch` server used to probe `localhost:*` ports

**Mitigation:**
- `mcp-policy.json` sets `allowNetwork: false` by default — network access governed at tool layer
- `fetch` MCP server is in `approvedServers` but subject to `defaultDeny: true` policy
- `PreToolUse` hook can block URLs matching internal IP ranges (`10.*`, `172.16.*`, `192.168.*`, `169.254.*`)
- Agent instructions explicitly forbid fetching cloud metadata endpoints

**Severity:** High

---

## 4. Path Traversal

**Threat:** Agent writes files outside intended directories via crafted paths.

**Attack Vector:**
- Write tool called with `../../etc/cron.d/malicious` path
- Hook script constructs file paths from untrusted input without sanitization
- MCP `filesystem` server used to access files outside allowed directories

**Mitigation:**
- `mcp-policy.json` sets `allowFileWrite: false` by default — write access governed at tool layer
- `filesystem` MCP server in `approvedServers` but scoped by its own configuration
- Storage policy invariant constrains new file creation to `/Volumes/External HD/Desenvolvimento/`
- `PreToolUse` hook blocks writes to protected paths (`~/.ssh`, `/etc`, system directories)

**Severity:** High

---

## 5. MCP Server Scope Creep

**Threat:** MCP servers gain access to tools or data beyond their intended scope over time.

**Attack Vector:**
- New MCP server added to `approvedServers` without review of its tool surface
- Existing server updates add dangerous tools (shell execution, network access)
- Tool permissions drift from documented policy as servers are reconfigured

**Mitigation:**
- `mcp-policy.json` uses `defaultDeny: true` — every server must be explicitly approved
- `approvedServers` list is version-controlled and reviewed in PRs
- `maxToolCallsPerTurn: 80` limits blast radius per session turn
- `toolTimeoutMs: 300000` (5 min) prevents runaway tool chains
- Constitution invariant `protected_invariants` includes MCP policy as a protected invariant

**Severity:** Medium

---

## 6. AI Attribution Leakage

**Threat:** AI-generated commits, PRs, or release notes contain attribution markers that misrepresent authorship.

**Attack Vector:**
- Agent adds `Co-Authored-By: Claude` or `Generated with [tool]` to commit messages
- PR description templates include AI attribution by default
- Release notes generated by agent include bot signatures
- Over time, git history accumulates AI co-authors, creating false provenance

**Mitigation:**
- Constitution invariant `no-ai-attribution` is a protected invariant (rank 1)
- Agent instructions explicitly forbid AI attribution in all git/GitHub artifacts
- Commit message hooks can be configured to reject commits containing known AI attribution patterns
- Author of record is always Lucas Santana — agents are tools, not contributors

**Severity:** Medium (integrity/reputation risk, not data loss)

---

## 7. PR Automation Bypass

**Threat:** Agent automates actions on PRs that should require human review — merging PRs with human comments, acting on another person's PR, or bypassing review requirements.

**Attack Vector:**
- Agent squash-merges a PR that has unresolved review comments from a human reviewer
- Agent creates and merges PRs on branches owned by other contributors
- Agent uses `--admin` flag to bypass branch protection
- Composite skills (e.g., `/ship`, `/merge-confidently`) auto-merge without checking for human reviewer blocks

**Mitigation:**
- Constitution invariant `pr-automation-halt` is a protected invariant (rank 1)
- Absolute rule: never automate any action on a PR with comments from another person, or any open PR authored by another person
- Bots (dependabot, renovate, coderabbit, greptile, sonar) are explicitly excluded from "another person"
- `pr-gate-sweep.sh` only squash-merges PRs that are clean-mergeable with zero failed/pending checks
- No `--admin` bypass, no `--no-verify`, no force-merges
- Orchestrator (or human) gates on review subagent exit state before advancing

**Severity:** Critical (can ship unreviewed code to production)

## Review Cadence

This threat model should be reviewed:
- Before each major harness release
- When new MCP servers are added to `approvedServers`
- When new hooks are wired to tool events
- After any security incident (P0/P1 → root-cause artifact committed before next task)
